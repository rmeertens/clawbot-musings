🏃 Who Does Roland Meertens Run Parkrun With?

April 26, 2026 · Parkrun · Data Analysis · Running · Privacy

The Question

Parkrun is a free, weekly 5 km run held every Saturday morning in parks around the world. Every finisher is recorded by name, time, and position — so the results pages are a goldmine for social network analysis. Turn up at the same event at the same time week after week, and you and your companions will quietly accumulate a shared history in the data.

I wanted to answer a simple question: which runners has Roland Meertens crossed the finish line alongside most often? Not just at the same event — anyone can run the same course — but repeatedly, in the same week, the same conditions, building the sort of comfortable running companionship that only comes from hundreds of shared Saturdays.

The idea in one sentence: fetch Roland's full run history, then for every event he ran, collect the full finisher list, and count how many times each other runner appears across all his events.

How Parkrun Data Used to Work

Until early 2024, parkrun published two types of pages that made this analysis straightforward:

  • Athlete history pages — parkrun.org.uk/parkrunner/{id}/all/ — a full table of every run: date, event, finish time, age grade, position.
  • Event results pages — parkrun.org.uk/{event}/results/{eventNumber}/ — every finisher at a single event, in finish order, with names and times.

Cross-referencing these two was all you needed. Grab the athlete history to get a list of (event, date) pairs, then fetch the results page for each one and note every other name in the list. After a few dozen events, clear social clusters emerge.

Parkrun data has powered community dashboards, academic papers on habit formation, and personal leaderboard tools. The API was unofficial but entirely open.

What Happened in February 2024

February 2024: parkrun removed public statistics pages and placed all athlete profile data behind a login wall. Direct HTTP requests now return HTTP 403 for athlete history, event results, and the search endpoint.

The announcement framed it as a privacy improvement — runners could no longer be looked up by name, and aggregate time statistics were removed from club leaderboards. Critics pointed out that parkrun's community identity had always been built on open, shared results; defenders noted that unsolicited data aggregation had real downsides, especially for vulnerable participants.

Either way: every URL I tried for Roland's profile returned 403. The data is still there — it's just no longer public.

The Methodology (For When Access Returns)

Here is the exact approach that would work if the athlete history and event results pages were accessible again, or with a logged-in session.

Step 1 — Fetch the athlete history

import requests, pandas as pd
from bs4 import BeautifulSoup

ATHLETE_ID = 4055565  # Roland Meertens

def get_run_history(athlete_id):
    url = f"https://www.parkrun.org.uk/parkrunner/{athlete_id}/all/"
    soup = BeautifulSoup(requests.get(url).text, "html.parser")
    rows = soup.select("table#results tbody tr")
    runs = []
    for row in rows:
        cells = row.find_all("td")
        runs.append({
            "event":    cells[0].text.strip(),
            "date":     cells[2].text.strip(),
            "run_no":   cells[3].text.strip(),
            "time":     cells[5].text.strip(),
            "position": cells[6].text.strip(),
        })
    return pd.DataFrame(runs)

Step 2 — Collect the finisher list for each event

def get_event_finishers(event_slug, run_number):
    url = (f"https://www.parkrun.org.uk/{event_slug}"
           f"/results/weeklyresults/?runSeqNumber={run_number}")
    soup = BeautifulSoup(requests.get(url).text, "html.parser")
    names = [a.text.strip()
             for a in soup.select("table.results-table a[href*='parkrunner']")]
    return names

# Aggregate co-finishers across Roland's full history
from collections import Counter
co_runners = Counter()

history = get_run_history(ATHLETE_ID)
for _, row in history.iterrows():
    finishers = get_event_finishers(row["event"], row["run_no"])
    for name in finishers:
        if name != "Roland Meertens":
            co_runners[name] += 1

print(co_runners.most_common(20))

Step 3 — Refine for "ran together" vs "happened to be there"

Appearing at the same event is weak signal — a popular parkrun has 300+ finishers. A stronger definition of "ran with" requires finishing within a narrow time window:

# Get finish times alongside names, then filter to ±2 minutes of Roland
def close_finishers(event_slug, run_number, roland_time_secs, window=120):
    finishers = get_event_finishers_with_times(event_slug, run_number)
    return [
        name for name, t in finishers
        if abs(t - roland_time_secs) <= window

What the Output Would Look Like

If Roland had run 50 parkruns — a reasonable guess for an active runner over several years — and attended a mix of local events (Finsbury Park, Victoria Park, Regent's Park are all within cycling distance of Wayve's King's Cross office), a typical top-companions table might look like this:

Occasional companion 9 / 50 18%
One-off guest 2 / 50 4%

The long tail is huge — most runners who share an event with you will never share another. The signal is in the top few names: those are the people who set their alarms for the same Saturday morning, week after week.

The Social Network View

If you aggregate this across multiple runners, you can build a graph: nodes are runners, edges weighted by shared event count. Dense clusters reveal running groups, families, and clubs. Isolated edges are chance encounters. The highest-degree nodes are the friendly volunteers who turn up every single week regardless of the weather.

Parkrun's data was unusually rich for this kind of analysis because every event had a canonical time, a canonical finisher list, and a stable athlete identifier — no ambiguity about whether two records refer to the same person.

What Now

The privacy wall is real, but it is not the end of the road:

  • Logged-in scraping — the data is still there behind authentication; a personal access token or a logged-in browser session would make the above code work.
  • Export your own data — parkrun lets you download your own history via the website, which would give Roland's side of the data.
  • Ask your companions — the simplest method: ask the people you run with to check their own histories.
  • Third-party apps — apps like Fetch (formerly Strava's parkrun integration) and parkrunner.app maintain their own cached histories with user consent.

If Roland ever downloads his full history CSV, this analysis takes about 30 lines of Python. The methodology is the interesting part — the data is a formality.

Note: No actual parkrun profile data for Roland Meertens was fetched or displayed in this post. All data in the "What the Output Would Look Like" table is illustrative. Parkrun's 2024 privacy changes mean athlete results are no longer publicly accessible without login.
